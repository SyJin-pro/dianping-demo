## 项目启动分析方法（从0到1）

在开始写代码前，先按以下顺序做分析：

1. 业务目标：第一阶段必须跑通哪些核心链路（登录、店铺浏览、互动）
2. 数据模型：核心实体关系、事务边界、索引策略
3. 非功能需求：性能、稳定性、安全性、可维护性
4. 阶段约束：当前以“最小可运行 + 可迭代”为优先

这套分析会直接决定技术选型，而不是先选技术再找场景。

---

## 第一阶段技术选型（MVP）

### 1) Web框架
- Spring Boot
- 原因：启动快、生态成熟、与 MySQL/Redis 集成稳定，适合快速搭建可运行骨架。

### 2) 数据存储
- MySQL（主存储）
- Redis（缓存 + 分布式Session）

### 3) 鉴权方案（当前确定）
- Option A：Session + Redis
- 原因：学习成本低、实现直观、支持多实例共享登录态，便于后续平滑演进到 JWT 等方案。

### 4) ORM与SQL策略
- MyBatis-Plus 处理基础 CRUD
- 手写 SQL 处理复杂查询（多表关联、聚合统计、复杂排序分页等）

> 说明：所谓“ORM复杂SQL”，是指自动生成SQL无法高效覆盖复杂业务查询时，需要手写SQL以保证性能和可控性。

### 5) 缓存策略（当前确定）
- Option A：Cache Aside（旁路缓存）为主
- 防护策略：空值缓存（防穿透）、互斥锁/逻辑过期（防击穿）、TTL随机化（防雪崩）
- 后续迭代：按业务热点引入更细粒度缓存治理

## 分层建议（第一阶段最小骨架）

当前仅创建 4 个核心包，先保证主链路可用：

- controller：接口层（接收请求、参数校验、统一返回）
- service：业务层（业务编排、事务边界）
- mapper：数据访问层（基础 CRUD + 后续复杂 SQL）
- entity：实体层（与数据库表结构对应）

> 说明：config、util、dto、vo 等包在实际需要时再补，避免前期过度设计。

# 第一阶段实现：
启动：
这行会启动 Spring 容器（ApplicationContext）和内嵌 Tomcat。 
1. 组件扫描与实例化
@SpringBootApplication 会触发扫描 com.jj.dianpingdemo 包及子包。
因为 UserController 上有 @RestController，Spring 会把它注册成 Bean。
同理 UserService（@Service）也会被创建，并通过构造器注入到 UserController。
2. 建立“URL -> 方法”映射
Spring MVC 在启动时读取 @GetMapping("/user/{id}")，建立路由表：
GET /user/{id} -> UserController#getUserById(Long id)。
3. 请求到来时的真实调用者
浏览器/客户端请求 GET /user/1
-> Tomcat 接收
-> DispatcherServlet（Spring MVC 前端控制器）处理
-> 找到匹配的 HandlerMethod
-> 通过反射调用 getUserById(1)
-> 你的方法里再调用 userService.getById(1)
-> 返回 User，由 @RestController 自动序列化成 JSON 返回。


@SpringBootApplication 负责“把组件装进容器”
@GetMapping + @PathVariable 负责“把请求路由到方法并绑定参数”
@Service 负责“业务组织”
@Mapper + @Select 负责“访问数据库”。

application.properties里配置数据库连接信息。
