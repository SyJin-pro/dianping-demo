# Spring Boot 拦截器【核心笔记・极简回顾版】
一、核心作用
在Controller 执行前做校验：登录验证、权限拦截、请求预处理。
二、配置固定 2 步
1. 写拦截器
   实现 HandlerInterceptor 接口，重写 preHandle（90% 场景只用这个）  
`   public class LoginInterceptor implements HandlerInterceptor {
   // 前置拦截：Controller执行前调用  
   @Override  
   public boolean preHandle(...) {  
   // 校验逻辑：未登录则拦截  
   return true; // true=放行 | false=拦截  
   }  
   }`
2. 注册拦截器
   @Configuration + WebMvcConfigurer 注册并配置路径  
`   @Configuration  
   public class MvcConfig implements WebMvcConfigurer {  
   @Override  
   public void addInterceptors(InterceptorRegistry registry) {  
   registry.addInterceptor(new LoginInterceptor())  
   .addPathPatterns("/**")        // 拦截所有  
   .excludePathPatterns(          // 放行接口  
   "/user/login", "/user/code"  
   );  
   }  
   }`
   三、关键规则  
   /**：拦截所有接口  
   /user/**：拦截 /user/ 下所有接口  
   preHandle return true → 放行进 Controller  
   preHandle return false → 直接拦截，不进 Controller  
   四、请求执行流程（必记）  
   浏览器请求 → DispatcherServlet（总入口） → 拦截器 preHandle → Controller  
   五、核心特点  
   拦截器是 Spring MVC 组件，只拦截 Controller 请求  
   由框架自动调用，无需手动触发  

# 登录校验拦截器 LoginInterceptor
1. 作用
   统一拦截需要登录的接口，校验用户登录状态，未登录直接拒绝。
2. 拦截规则
   拦截：/** 所有接口
   放行：/user/login、/user/code、/user/logout（无需登录）
3. 核心逻辑
   检查 Session 中是否有登录用户
   无用户 → 拦截，返回未登录
   有用户 → 存入UserHolder，放行到 Controller
   请求结束 → 清空用户上下文
4. 核心价值
   统一校验、保护接口、代码简洁、全局获取用户


## 极简笔记（双拦截器总结）
1. 执行顺序（绝对正确）
   先刷新 → 后登录order=0 先执行，order=1 后执行
2. 职责拆分（绝对正确）
   RefreshTokenInterceptor：全局刷新、存用户、不拦截
   LoginInterceptor：仅校验登录、拦截未登录请求
3. 路径规则（修正后）
   刷新：拦截所有 /**
   登录：只放行登录 / 验证码，拦截其余所有
4. 核心价值
   无状态 Token 登录（前后端分离标准）
   自动刷新会话有效期
   统一登录校验，Controller 无冗余代码

## 问题
为什么第一个拦截器把查询到的用户存到Threadlocal？为什么不就在redis中查询，redis也是基于缓存的呀。  
答：
ThreadLocal 存用户的核心意义
1. 请求级本地缓存：比 Redis 更快，避免重复查询 Redis
2. 全链路复用：一次查询，拦截器 / Controller/Service 通用
3. 解耦职责：双拦截器、业务代码之间无缝传递用户
4. 线程安全：每个请求独立，用户信息不混乱  

ThreadLocal会被保存多久？  
ThreadLocal 里的用户数据，只存活【当前这一次 HTTP 请求】！
请求一结束，数据立刻被销毁，绝对不会长期保存！  
ThreadLocal不是天然具有线程隔离的优点吗？
线程隔离是天生的，防的是「同时多人」；
remove () 是手动加的，防的是「线程复用串数据」。
