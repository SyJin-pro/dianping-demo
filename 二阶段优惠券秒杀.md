## 数据库表
tb_voucher: 优惠券表  
tb_seckill_voucher: 秒杀优惠券表
tb_voucher_order: 优惠券订单表  
## 业务流程
1. 用户领取优惠券，系统将优惠券信息保存到tb_voucher_order表中，状态为未使用
2. 用户在结算时使用优惠券，系统将优惠券状态更新为已使用，并将订单信息保存到tb_voucher_order表中
3. 系统定期清理过期的优惠券订单，将状态为未使用且过期的订单删除
4. 系统定期清理过期的优惠券，将状态为未使用且过期的优惠券删除

   五、验证并发漏洞
   执行以下 SQL 验证结果：
   sql
   -- 1. 查看订单：同一用户（如 user_id=1012）会出现多条订单记录（一人一单失效）
   SELECT * FROM tb_voucher_order WHERE voucher_id = 11;

-- 2. 查看库存：库存为 0（因扣库存 SQL 带 `WHERE stock > 0`，不会出现负数）
SELECT stock FROM tb_seckill_voucher WHERE voucher_id = 11;
核心现象
✅ 一人一单失效：同一个用户在并发下抢到了 N 张券（本应只能抢 1 张）
✅ 库存安全：库存不会扣成负数（由 SQL 条件 stock > 0 兜底）
六、关键踩坑提醒
变量语法：JMeter 变量用 ${变量名}，不是 Postman 的 {{变量名}}
组件层级：JSON 提取器必须是登录接口的子节点，确保 Token 能传递到当前线程的秒杀请求
请求头格式：登录接口用表单请求头 application/x-www-form-urlencoded，秒杀接口用 application/json
验证码关闭：必须完整注释后端验证码校验逻辑，否则登录会失败
七、本质结论
基础版秒杀的 **「查订单是否存在」和「创建订单」两步操作非原子性 **，高并发下多个线程会同时通过「一人一单」校验，最终导致同一个用户创建多条订单，业务规则完全失效。
